# Plan to Fix Sales Visualization in Forge

## Root Cause Analysis
The issue stems from how `forge` manages tool inputs and context for MCP Apps widgets, which forces the LLM to hallucinate data:

1. **LLM Hallucination**: Because `forge` passes the strict tool schema to the LLM and the `visualize-sales-data` tool requires a massive `report` object, the LLM (Gemini 2.5 Flash) takes a shortcut and hallucinates a fake, small `report` object instead of copying the real one.
2. **Immediate "Hardcoded" Rendering**: When the LLM calls the tool with hallucinated data, `forge`'s bridge immediately sends this fake data to the iframe via `ui/notifications/tool-input`. The iframe blindly renders it, making the visuals look hardcoded.
3. **No Server Logs**: The hallucinated `report` object is invalid according to the server's strict Zod schema. The server throws a validation error before the tool handler executes, which means `console.log` is never called. The `mcpClient.callTool` fails, but because of how `forge` handles the fallback, the widget still renders the fake data.

## Proposed Fix
Since we cannot modify the `sample-mcp-apps-chatflow` server code (which works correctly on other MCP clients), we must fix how `forge` passes data to the embedded MCP Apps.

1. **Update `useBridgeMessages.js`**:
   We will update `buildToolInputArguments` to prioritize the actual data generated by previous tools (which is available in the context) over the hallucinated arguments from the LLM. If the `structuredContent` from a previous step (or the current tool output) contains the valid `report`, we inject it so the UI receives the real data instead of the fake data.
   
2. **Handle Context Properly in `McpAppsPanel.jsx` / `aiChatService.js`**:
   We will ensure that when `ui/update-model-context` is called, the data is properly persisted and made available to subsequent widget initializations so that `sales-visualization.ts` can read the real `report` data directly from the bridge.

By fixing the bridge implementation in `forge`, the iframe will receive the authentic data, bypass the LLM's hallucinations, and render the actual visualization without requiring any changes to the server code.
