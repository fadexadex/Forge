# Plan: Fix Forge App Tool Execution for Complex Inputs

## Issue Analysis
The user reported that when making a request to the `visualize-sales-data` tool within the Forge app (test workbench), they "don't get a response" (the UI likely hangs in a loading state) and suspect it's due to the complex inputs they are sending. The feedback specifically states that we cannot modify the `visualize-sales-data` tool itself, because other MCP clients handle it perfectly.

Upon inspecting `src/stores/testStore.js` in the Forge app, I found three critical issues causing this:
1. **Unhandled Promise Rejections (Hangs):** In the `EXTERNAL MODE` execution path for tools, there is no `try...catch` block around `await client.callTool(...)`. If the user submits inputs that fail Zod validation on the server (e.g., missing required fields, or a string where an object is expected), the server returns a JSON-RPC error. The MCP SDK throws this as an `McpError`, which crashes `executeTool`, leaving the UI stuck in `isExecuting: true` forever.
2. **Brittle Input Parsing:** The Forge app tries to parse form textarea strings into JSON objects only if `schema.type === 'array' || schema.type === 'object'`. However, complex schemas generated by Zod might omit a direct `type` property (using `$ref`, `anyOf`, etc.). If `type` is missing, the app sends a raw string instead of a JSON object to the server, immediately causing a validation error and triggering the hang described in #1.
3. **Hidden `structuredContent`:** Even if the tool succeeds, `testStore.js` sets `response.data = toolResult.content`, completely ignoring `toolResult.structuredContent` and `toolResult._meta`. This means the user never sees the rich structured data returned by the visualization tool in the OutputPanel.

## Proposed Solution
We will fix the Forge app (`src/stores/testStore.js`) to correctly handle tool execution, validation errors, and rich responses.

### Targeted File
`src/stores/testStore.js`

### Implementation Steps
1. **Add `try...catch` for Tool Execution:**
   Wrap the `client.callTool` call inside a `try...catch` block. If an error is caught, cleanly update the store's state with `response = { success: false, data: null, error: err.message, responseTime }` so the UI shows the validation error instead of hanging.
2. **Enhance Input Parsing:**
   Improve the `JSON.parse` check for form inputs. Instead of relying strictly on `schema.type`, check if the string value appears to be JSON (`val.trim().startsWith('{')` or `val.trim().startsWith('[')`). This ensures complex objects and arrays like `selections` and `report` are always properly converted to objects before sending.
3. **Include Full Tool Result:**
   Change the successful response assignment so that `data` includes the full `toolResult` (or at least `content`, `structuredContent`, and `_meta`) rather than just `toolResult.content`. This ensures the OutputPanel correctly displays all the data returned by the tool, exactly matching the structure the user saw in the other MCP app tester.
